## 1. Introduction

Recently, many enterprises have been moving toward adopting blockchain. However, one of the biggest obstacles to mainstream adoption is the **finality speed problem**. Even after a transaction has been submitted, there is a delay before it is finally confirmed, which makes blockchain difficult to apply to real-world services that require immediate responsiveness.

To improve this, various consensus algorithms have been studied, and **CometBFT**, the flagship consensus engine of the Cosmos ecosystem, is one of them.

Yet, if you dive deep into the CometBFT code, you can find an interesting piece of logic that **intentionally delays consensus**. In this post, I want to examine why this code exists and what problems it causes, and then think concretely about how we can improve consensus speed without harming the fairness of the ecosystem.

---

## 2. Tendermint & CometBFT

To deeply understand the consensus structure of **CometBFT**, we first need to understand **Tendermint**, its root.

At its core, Tendermint is a consensus algorithm that **adapts the classic distributed-systems problem known as the Byzantine Generals Problem to the blockchain environment**.

Even if some nodes are maliciously manipulated and send false messages, the whole system must still reach the same consensus state. Consensus algorithms that can tolerate such Byzantine faults and still maintain a healthy network are called **BFT (Byzantine Fault Tolerance)**.

Earlier algorithms like PBFT (Practical BFT) were limited to small closed networks, but Tendermint improved on this so that many nodes can participate and quickly and safely reach consensus — especially in public, Proof-of-Stake (PoS) environments.

The most notable characteristic of the Tendermint consensus algorithm is that **once more than 2/3 of the total validator stake agrees, the block is finalized immediately — i.e., it guarantees Instant Finality (deterministic finality)**.

"Tendermint Core" bundles this BFT consensus algorithm with P2P networking, and to communicate with the application layer it introduced ABCI (Application Blockchain Interface).

### 2-1. Tendermint Consensus Process

Tendermint describes the process of overcoming Byzantine faults and achieving instant finality mathematically, modeling it as a **state machine**. The diagram below shows how nodes transition voting states within a single round.

The consensus process is broadly split into a voting Round in which a block is proposed and voted on, and a Commit step that finalizes the block after consensus is reached. To defend against Byzantine nodes, each voting step requires more than 2/3 of the total validator stake to agree before proceeding.

1. **Propose**: The round's Proposer (leader) proposes a new block. If the block doesn't arrive in time or is invalid, the flow falls into the `Prevote Nil` path.
2. **Prevote**: If the block is valid, nodes vote `Prevote Block`; otherwise they vote `Prevote Nil`. Nodes wait for other nodes' votes and proceed to the next step once more than 2/3 are collected.
3. **Precommit**: Nodes that have confirmed more than 2/3 of prevotes cast `Precommit Block`, signaling the final intent to record the block on chain. If a 2/3 majority cannot be gathered here, a New Round is started to prevent the system from halting.
4. **Commit and New Height**: Once more than 2/3 of Precommit votes are obtained for a specific block, the block is finally committed. The block is then stored in the local DB, and the chain advances to the next block height (New Height) to start stacking a new block.

### 2-2. CometBFT Consensus Process

**CometBFT**, the current standard consensus engine of the Cosmos ecosystem, is a successor project that was officially forked from Tendermint Core in 2023 and is now developed under the stewardship of the ecosystem.

One of the biggest differentiators of CometBFT compared to Tendermint is the introduction of **ABCI++**, an extended communication interface with the application. In the past, Tendermint could only hand a block to the application layer (Cosmos SDK) after consensus was fully complete, but CometBFT provides extension points for the application to intervene at every step: `PrepareProposal` before proposing a block, `ProcessProposal` just after receiving a proposal, and even `VoteExtensions` during the voting process.

This enables much more sophisticated work at the consensus layer, such as MEV (Miner Extractable Value) mitigation and oracle data inclusion.

For this three-step consensus process to keep running without stalling, various "timeout" mechanisms are essential to guard against network delays or node failures.

Yet, if you look inside CometBFT, you can find logic that **does not merely guard against faults but intentionally delays block production even after 2/3 agreement has already been reached and consensus is effectively complete**.

## 3. Timeout commit & Missing

### 3-1. timeout_commit

CometBFT has a configuration value called `timeout_commit`. This is an additional waiting time held at the Precommit step, even after the 2/3 votes required to finalize a block have already been collected.

If you look at the actual code, you can see that inside the logic that runs once more than 2/3 of Precommit votes have been gathered — i.e., once consensus is finalized — the engine intentionally waits for the configured `timeout_commit` duration.

```go:Go  cometbft/consensus/state.go {13-14}
func (cs *State) enterNewRound(height int64, round int32) {
	// ... (omitted) ...

	switch cs.Step {
	case cstypes.RoundStepNewHeight: // timeoutCommit phase
		if cs.needProofBlock(cs.Height) {
			// enterPropose will be called by enterNewRound
			return
		}

		// Even though we have gathered the 2/3 signatures needed to finalize the block
		// and reached the NewHeight step, instead of advancing to the next round (NewRound)
		// immediately, we schedule a wait for the configured timeout_commit duration.
		timeoutCommit := cs.StartTime.Sub(cmttime.Now()) + 1*time.Millisecond
		cs.scheduleTimeout(timeoutCommit, cs.Height, 0, cstypes.RoundStepNewRound)

	// ... (omitted) ...
}
```

> [state.go GitHub source](https://github.com/cometbft/cometbft/blob/main/consensus/state.go#L1050-L1058)

The scheduled timeout is then processed by an internal timer routine. The duration passed into `scheduleTimeout` is delivered through the `tickChan` channel, which starts the timer. Once the timer expires and a signal is received on `timer.C`, the consensus engine finally receives a `tockChan` signal telling it to proceed to the next step.

```go:Go  cometbft/consensus/ticker.go
func (t *timeoutTicker) timeoutRoutine() {
	t.Logger.Debug("Starting timeout routine")
	var ti timeoutInfo
	for {
		select {
		// 1. When scheduleTimeout is called from the earlier state.go, the signal arrives on tickChan
		case newti := <-t.tickChan:
			// ... (omitted) ...
			// Set the timer with the received timeout_commit duration (ti.Duration)
			ti = newti
			t.timer.Reset(ti.Duration)
			t.timerActive = true

			t.Logger.Debug("Scheduled timeout", "dur", ti.Duration, "height", ti.Height, "round", ti.Round, "step", ti.Step)

		// 2. Once the configured duration (timeout_commit) has fully elapsed, the timer fires
		case <-t.timer.C:
			t.timerActive = false
			t.Logger.Info("Timed out", "dur", ti.Duration, "height", ti.Height, "round", ti.Round, "step", ti.Step)

			// 3. Finally notify the consensus state machine that the timeout has expired,
			//    so it can move on to the next step (NewRound).
			go func(toi timeoutInfo) { t.tockChan <- toi }(ti)

		case <-t.Quit():
			t.stopTimer()
			return
		}
	}
}
```

> [ticker.go GitHub source](https://github.com/cometbft/cometbft/blob/d9519e1a012c3d7bbd85f16b48113d6185ac5456/consensus/ticker.go#L96-L140)

Even though all the votes needed to finalize the block have been gathered, the timer routine keeps running for the configured `timeout_commit` duration and intentionally delays the block decision.<br/>
**_As you can see from the code, this is not some strictly necessary piece of work — it simply waits as configured._**

After thinking about why this exists, I concluded that without a `timeout_commit`, as soon as 2/3 of the votes arrive the engine would immediately advance to a new height and proceed to the next propose. In that case, due to physical distance between nodes, packet loss, or other network issues, some validators' votes could arrive late, leading to unfairly penalized validators.<br/>
This seems to be the reason `timeout_commit` was introduced in the first place.

Indeed, if you look at issue #2525 in the Cosmos SDK repository, there was a design discussion about whether to "apply micro-slashing (slightly reduced rewards for validators that voted late)" or "wait long enough so that everyone can receive rewards". In other words, the core question was: how should the system rescue validators that might otherwise unfairly miss out on rewards?

> **Related issue discussion:** [cosmos-sdk/issues/2525](https://github.com/cosmos/cosmos-sdk/issues/2525#issuecomment-430838701)

However, looking at that issue, only the discussion has taken place. No clear consensus or code has been applied yet, and the actual implementation still contains a `TODO`.

Given that this still isn't implemented, I think one reason the consensus engine still holds on to this waiting period is the validators' **Missing Count and Jail** system.

### 3-2. Missing Count

**Missing Count**, as the name suggests, is a counter that increases when a validator fails to vote in time. The network checks, within a configured **Block Window**, whether a given validator has exceeded the allowed Missing Count (`Max Missed`). If so, the validator is sent to Jail and its participation in the network is restricted.

<Tip>
**How to compute Max Missed**<br/>
`maxMissed = SignedBlocksWindow - MinSignedPerWindow`<br/>
(maximum allowed misses = window size - minimum blocks that must be signed)

</Tip>

```go:Go  cosmos-sdk/x/slashing/keeper/infractions.go
    ...
    }
```

> [infractions.go GitHub source](https://github.com/cosmos/cosmos-sdk/blob/d4bd475869271dbe6cccd77458b4bcaaf4a0669e/x/slashing/keeper/infractions.go#L109-L113)

The actual Missing Count handling logic looks like this:

```go:Go  cosmos-sdk/x/slashing/keeper/infractions.go
// handleValidatorSignature: checks each validator's signature status and updates the bitmap and counter
func (k Keeper) handleValidatorSignature(ctx context.Context, addr crypto.Address, power int64, signed BlockIDFlag) error {
	// ... (omitted) ...

	// Check whether the validator signed (voted) this block
	missed := signed == types.ValidatorAbsent

	// If the validator failed to sign
	if missed {
		// Increment MissedBlocksCounter by 1 (if this accumulates, the validator goes to Jail)
		signInfo.MissedBlocksCounter++
		// Also mark as missed in the bitmap
		err := k.SetMissedBlockBitmapValue(ctx, addr, signInfo.IndexOffset, true)
	} else {
		// ... normal signing logic ...
	}
    // ...
}
```

As you can see, once signature info arrives as `ValidatorAbsent`, `missed = true` is set and `MissedBlocksCounter` is immediately incremented, while `SetMissedBlockBitmapValue` records the vote-miss for that block.

Because a blockchain is a distributed network of globally scattered nodes communicating over the Gossip protocol, **long-tail** situations are inevitable — depending on the physical distance between nodes or network routing, a particular node's Vote packet can easily arrive 0.1–0.5 seconds late.

However, if the chain advanced to the next block as soon as 2/3 consensus was reached, slightly-late votes would all be discarded. The corresponding validators would then be treated as having missed the vote and unfairly penalized.<br />
As a result, `timeout_commit` effectively plays the role of a "grace period" that absorbs such network delays and prevents honest nodes' votes from being dropped and penalized.

### 3-3. Limits of timeout_commit and how the discussion evolved

Ultimately, if you completely remove `timeout_commit`, late-arriving votes will be ignored and honest validators will frequently end up in Jail, so it is hard to remove it outright.

But there are clear shortcomings. It is a local value that each validator has to configure individually, and once the node is running it is a static value that doesn't change. People have repeatedly pointed out that this is poor from a UX standpoint.

This problem was already raised in Tendermint back in 2021, and was raised again in CometBFT in 2024.<br/>
Eventually, the direction was to deprecate `timeout_commit` and control this behavior through a new parameter named `next_block_delay`.

> **Related discussion issues**
>
> - [Tendermint Issue #5911](https://github.com/tendermint/tendermint/issues/5911)
> - [CometBFT Issue #2655](https://github.com/cometbft/cometbft/issues/2655)

In 2024 this discussion was finalized through **ADR 115**, which aims to make block times more predictable.

> [**ADR 115: Predictable Block Times**](https://github.com/cometbft/cometbft/blob/main/docs/references/architecture/adr-115-predictable-block-times.md)

However, the primary goal of ADR 115 is to keep the interval between blocks consistent; it is not meant to make consensus itself faster.

It's not that there have been no attempts to speed things up. As the Cosmos ecosystem has strengthened PoA support to meet enterprise demand, a `skip_timeout_commit` option was added that allows skipping the configured waiting time.

> [Running in Production Guide](https://github.com/cometbft/cometbft/blob/dfcc41ca42076a037b382984f77d7723f8f38e00/docs/core/running-in-production.md?plain=1#L368-L375)

However, as the official guide clearly states, this feature is only recommended for PoA environments where economic penalties do not apply. In a public PoS environment it can cause unfair slashing, and the guide strongly recommends keeping the option disabled.

Putting all these open-source discussions together, the current solutions may be well suited to improving UX by keeping block intervals uniform, but **they still do not fundamentally improve consensus speed in a public environment while also protecting validators from unfair slashing**.

## 4. Improvement Directions

So I started thinking about what structural alternatives could remove the `timeout_commit` waiting period in a public PoS environment to speed up consensus, while still preventing validators from being unfairly penalized by network delays.

The key idea is to **defer the point at which Missing Count is evaluated and updated — from the current `n-1` block to `n-2` or earlier**. This allows late-arriving votes to still be counted as valid network participation.

### 4-1. Deferring the Last Commit Evaluation

The first approach is to change the evaluation basis of `LastCommit` — which is checked at Finalize Block time to determine validator commit status and update Missing Count — from the `n-1` block to an `n-2` or earlier block.

The biggest advantage of this approach is that it does not require major changes to the Cosmos SDK or the ABCI layer — it can be implemented simply by modifying CometBFT internally so that it collects the commit for the `n-2` block.

Validator nodes may need a little more memory, but we can shorten block time with minimal architectural changes.

However, this approach has a side effect that conflicts with the **block-time computation algorithm**.<br/>
In the consensus engine, a new block's timestamp does not follow any central server clock. Instead, as shown below, it is derived as the `Median Time` of the vote timestamps from the previous block's participants included in `LastCommit`, and that median is used as the new block's official time.<br/><br/>

```go:Go  cometbft/state/state.go
// The actual CometBFT BFT Time (Median Time) computation logic
func (commit *Commit) MedianTime(validators *ValidatorSet) time.Time {
	weightedTimes := make([]*cmttime.WeightedTime, len(commit.Signatures))
	totalVotingPower := int64(0)

	for i, commitSig := range commit.Signatures {
		if commitSig.BlockIDFlag == BlockIDFlagAbsent {
			continue // Exclude Absent
		}
		_, validator := validators.GetByAddress(commitSig.ValidatorAddress)
		if validator != nil {
			totalVotingPower += validator.VotingPower
			// Collect timestamps, weighted by each voter's stake (VotingPower)
			weightedTimes[i] = cmttime.NewWeightedTime(commitSig.Timestamp, validator.VotingPower)
		}
	}

	// Return the time corresponding to the median of the collected, stake-weighted timestamps
	return cmttime.WeightedMedian(weightedTimes, totalVotingPower)
}
```

> [MedianTime GitHub source](https://github.com/cometbft/cometbft/blob/main/state/state.go#L274-L293)

If we change `LastCommit` to be based on the `n-2` block instead of `n-1` just to defer Missing Count, then the time of the currently-being-produced block would be computed from timestamps "two blocks ago" instead of from the previous block — a critical time-error issue.

It also complicates genesis-time handling. Previously, the first block could have finality derived directly from the genesis information, but if the basis is pushed back by one step to "two blocks ago", there will be a gap in proving finality for the first few blocks at network bootstrap, or we would be forced to add extra special-case logic.

In conclusion, even though this looks like a simple change to the evaluation basis, it triggers unexpected cascading architectural changes ranging from block-time computation to early finality proofs, so it must be approached very carefully.

### 4-2. Adding a Delayed Commit Data Field

The second approach is to keep the existing `LastCommit` untouched — continuing to use it for block-time computation and immediate consensus — and instead add a new data field to the Cosmos SDK slashing module to record delayed votes separately.

First, let's look at how the slashing module currently manages validator state. In the actual code, the slashing module strictly separates the heavy bitmap DB from the simple cumulative counter. If we just look at the immediate vote information coming from the consensus engine and carelessly mark failures in this bitmap, rolling back later to undo unfair slashing becomes architecturally extremely tricky.

```go:Go  cosmos-sdk/x/slashing/types/slashing.pb.go
// 1. Cumulative counter (in memory)
type ValidatorSigningInfo struct {
	Address             string
	StartHeight         int64
	IndexOffset         int64     // offset used to compute bitmap position
	MissedBlocksCounter int64     // simple cumulative miss count (used for Jail decisions)
    // ...
}

// 2. Missed-block bitmap (stored in the KVStore in chunks)
// Key: ValidatorMissedBlockBitmapPrefix + Address + ChunkIndex
// Value: 1024-bit Array (1=Missed, 0=Signed)
//
// [Chunk 0] 001011000100... (bits 0 ~ 1023)
```

> [slashing.pb.go GitHub source](https://github.com/cosmos/cosmos-sdk/blob/main/x/slashing/types/slashing.pb.go)

Therefore, to avoid damaging this existing state-management structure, we need to design a mechanism that extends the ABCI layer — which connects the consensus engine and the application — to carry DelayedCommit. Concretely, ABCI++'s `RequestFinalizeBlock` needs to be extended as follows.

```protobuf:Protobuf  proto/tendermint/abci/types.proto
// ABCI extension
message RequestFinalizeBlock {
    repeated bytes txs = 1;

    // 1. Keep the existing field for the previous block N-1's valid votes,
    //    since it is still needed for existing logic including Median Time.
    CommitInfo decided_last_commit = 2;

    // 2. Add a new field for votes that were sent as Precommit at consensus time for block N-2
    //    but didn't arrive in time during block production.
    CommitInfo delayed_commits = 9;

    // ...
}
```

> [types.proto GitHub source](https://github.com/cometbft/cometbft/blob/main/proto/tendermint/abci/types.proto)

Once the slashing module can receive `DelayedVotes`, the remaining task is to design a deferred-update mechanism that postpones the point at which vote misses are judged.

Here is how Cosmos SDK currently processes vote misses:

```go:Go  cosmos-sdk/x/slashing/keeper/infractions.go
// Logic that only looks at the previous block N-1's votes and processes them immediately
func (k Keeper) HandleValidatorSignature(ctx context.Context, addr cryptotypes.Address, power int64, signed comet.BlockIDFlag) error {
	// ...
	missed := (signed == comet.BlockIDFlagAbsent)

	if missed && !previous {
		// Immediately increment the counter and update the bitmap
		k.SetMissedBlockBitmapValue(ctx, consAddr, index, true)
		signInfo.MissedBlocksCounter++
	}
	// ...
}
```

As you can see, if a vote hasn't arrived by a particular block, the counter is incremented and the bitmap is updated immediately.
If we tried to improve this by later rolling the state back, we would likely trigger state-inconsistency issues across the system.

So instead, I want to apply a Deferred Evaluation mechanism that pushes the judgement point back by one beat.

```go:Go  cosmos-sdk/x/slashing/keeper/infractions.go
// Add a DelayedVotes field carrying info about Precommits that were cast for block N-2's consensus
// but did not arrive in time.
func (k Keeper) HandleValidatorSignature(ctx context.Context, delayedCommits VoteInfos) error {

	if !delayedCommits.Contains(Validator) {
		// Finalize the vote miss for block N-2: increment the counter and record it in the bitmap
		k.SetMissedBlockBitmapValue(ctx, consAddr, index-1, true)
		signInfo.MissedBlocksCounter++
	}

}
```

> [infractions.go GitHub source](https://github.com/cosmos/cosmos-sdk/blob/main/x/slashing/keeper/infractions.go)

Then, during the next block's `FinalizeBlock`, we receive the `DelayedVotes` data and finalize our judgement on block N-2's votes.<br/>
If the vote still hasn't arrived even by this point, only then do we increment the Missing Count and update the bitmap. If the vote did eventually arrive late, the counter simply doesn't increment, and the vote is naturally recognized as a proper Commit.

By designing this deferred evaluation mechanism, we avoid unnecessary state rollbacks and don't touch the existing block-time computation logic at all, minimizing side effects on the consensus engine.<br/>
I think this is a safer direction that preserves structural separation between system components while solving the problem.

## 5. Conclusion

I have shared this content with the core team of our academic society, and we plan to move forward with detailed architectural design and technical verification focused on the second approach — introducing a DelayedCommit field and applying deferred updates — because it minimizes side effects and is structurally safer.

A blockchain is not just a piece of distributed software; it is a system in which participants' incentives and penalties — their economic interests — are tightly intertwined.
Through this research I was able to deeply appreciate just how tight the dilemma between maximizing network performance (block-time reduction) and preserving ecosystem fairness (protecting honest validators) really is.

Simply tweaking a single consensus-engine wait parameter was not enough to shorten block time. I needed to design a macro-level architecture that organically connected every layer of the system — from the lower-level physical communication delays in the P2P network, to the consensus-engine state machine, to ABCI extensions, and all the way up to the slashing/penalty logic in the Cosmos SDK.

I started from the narrow goal of reducing block time, but along the way I discovered the economic-incentive dilemmas and distributed-systems design intentions deeply intertwined with the consensus engine, and I was able to broaden my perspective on core-level architecture.<br/>

I believe that facing such design dilemmas is the closest path to learning the philosophy of open source, and I plan to keep thinking hard about these kinds of fundamental structural problems in core technologies going forward.
