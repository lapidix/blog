# 1. Introduction

There are several ecosystems in the blockchain space. Well-known networks like Bitcoin, Ethereum, and Solana can each be considered an ecosystem, but Cosmos is a bit unique.  
Cosmos provides an SDK and a protocol module called IBC (Inter-Blockchain Communication), supporting an environment where different networks equipped with this module can communicate with each other.  
Recently, while developing monitoring for IBC Transactions at DLV Labs, I discovered a bug in the packet timeout verification logic within IBC Transactions.

This gave me a desire to learn more about IBC in detail while contributing to open source. This post summarizes the process of discovering the bug, analyzing its cause, building a PoC, and making an open-source contribution over a period of more than a month!

# 2. IBC?

Simply put, IBC (Inter-Blockchain Communication) is a communication protocol between blockchains.
Fundamentally, each chain is independent. However, out of necessity, you might need to send Ethereum's ETH to Solana or a chain like Osmosis. Or, there might be cases where you need to read data from another chain.  
IBC is a standard specification for such cross-chain communication.

As shown in the picture above, because each chain has a built-in Light Client of the counterparty chain, it independently verifies whether the received information is correct. Since the Light Client of each chain performs this verification, even if a Relayer is malicious, it does not cause a problem.  
The Relayer merely plays the role of delivering messages, and if it delivers incorrect data, it is rejected during the verification phase.
Thanks to this structure, IBC is said to have never been hacked at the protocol level since its launch in 2021.

## 2-1. IBC Packet?

The basic unit of data transfer in IBC is a packet. The packet delivery process operates in a callback manner where the application logic of the sending chain and receiving chain is called at each step. Once the modules of the two chains are connected, the Relayer starts relaying packets and Acknowledgements bi-directionally.

### Normal Flow

![](/static/images/ibc-timeout-height-bug/packet_flow.png)

When a user sends a packet through the IBC App layer of Chain A, it is committed to the IBC Core. A Relayer detects this, forwards it to the IBC Core of the counterparty chain, and if it passes the Light Client verification on Chain B, it is successfully received.

Afterward, Chain B generates an Acknowledgement and sends it back to Chain A via the Relayer. Finally, when Chain A receives the Acknowledgement, both chains update their respective states, and the packet delivery is complete.

### Abnormal Flow

![](/static/images/ibc-timeout-height-bug/packet_timeout_flow.png)

The abnormal flow also starts the same way: a user sends a packet through the IBC App layer of Chain A, and it is committed to the IBC Core.  
However, if the packet is not delivered to Chain B due to a network failure or a Relayer issue, the Relayer checks the unreceived status of the packet on Chain B. During this process, it also checks for timeouts.
Timeouts can be set in two ways.

One is an absolute time in nanoseconds called **timestamp**, and the other is **height**. The Height has the format `<Counterparty Chain's Revision_number>-<Timeout Block Height of Counterparty Chain>`. Here, the Revision number is an identifier that increments every time the chain undergoes a hard fork or major upgrade resulting in a protocol version change. Since the block height can be reset after an upgrade, using it together with the Revision number allows representing a unique point in the chain's entire history.
For example, "Block 10000 of Cosmos Hub v14" and "Block 10000 of v15" are different points in time, so they are distinguished as `14-10000` and `15-10000`.

Timeout handling proceeds as follows.  
As the Relayer periodically checks the state of Chain B, if Chain B's current time has exceeded `timeout_timestamp` or Chain B's current height has exceeded `timeout_height`—that is, if either condition is met—it determines that a timeout has occurred. At this point, the Relayer delivers a TimeoutPacket to Chain A, and Chain A's OnTimeoutPacket callback is invoked to rollback the packet.

For example, in an IBC token transfer, the temporarily escrowed tokens are returned to the sender. Conversely, if neither condition is met, the Relayer continues attempting to deliver the packet.
This means that if the timeout conditions are configured incorrectly, a packet could remain in a pending state permanently.

> Detailed information can be found in the [official documentation](https://ibc.cosmos.network/v10/ibc/apps/ibcmodule/#packet-callbacks).

## 3-1 Defining the Problem

The current situation can be summarized as follows.

The `packet timeout height` generated in SendPacket of an IBC Transfer should be in the format `<Counterparty chain revision number>-<Timeout block height of counterparty chain>`. However, in the case of the abnormal transaction, it was being generated in the format `<Processed height on own chain>-<Processed height on own chain + 100>`.

If this fact is exploited, it becomes possible to create packets that will never time out.  
In normal IBC communication, a timeout is a safety mechanism that operates when a packet is not delivered due to network failure or a Relayer issue. It plays the role of preventing asset loss by rolling back tokens that were escrowed (temporarily held) on the sending chain.

However, if the revision number is set abnormally high, this safety mechanism does not operate.  
For example, even in a situation where the packet is actually not delivered due to network failure, if the timeout height is set to a value like `9999-100`, the timeout will not occur. We would have to wait until the counterparty chain's Revision number upgrades to 9999, which is practically a point in time that will never come.
As a result, the tokens escrowed on the sending chain are not rolled back, and they become permanently locked without being received by the receiving chain either.

Furthermore, the cost of the attack is almost zero. Since one simply needs to set an incorrect timeout height, the only requirement is the transaction fee, meaning that the network's safety mechanism can be bypassed without large funds or complex technology.

**I felt that such packets that never time out could have a negative impact on the network.**

So I started looking for why such cases arise.

## 3-2 Why would such an invalid packet be processed normally?

First, to find out why such an invalid packet is processed normally, I checked the code in the `ibc-go` repository.

Looking inside the `SendPacket()` function in `modules/core/04-channel/keeper/packet.go`, there was logic that creates a Timeout and then checks if the packet timeout is valid through `Elapsed()`.


Then let's take a closer look at the `Elapsed()` function in `modules/core/04-channel/types/timeout.go`.

```go:Go  modules/core/04-channel/types/timeout.go {4,10}
// Elapsed returns true if either the provided height or timestamp is past the
// respective absolute timeout values.
func (t Timeout) Elapsed(height clienttypes.Height, timestamp uint64) bool {
	return t.heightElapsed(height) || t.timestampElapsed(timestamp)
}

// heightElapsed returns true if the timeout height is non empty
// and the timeout height is greater than or equal to the relative height.
func (t Timeout) heightElapsed(height clienttypes.Height) bool {
	return !t.Height.IsZero() && height.GTE(t.Height)
}
```

The logic itself is simple. It checks if the Height has elapsed, and if not, it checks the timestamp. If neither has elapsed, it returns `false`, and if either one has elapsed, it returns `true`.

Since the problem we found is in the height verification part, we need to look deeper into the `heightElapsed()` method. The Height must not be zero, and the current height must not be greater than or equal to the timeout height. Here, GTE stands for Greater Than or Equal to, which is a method to compare if something is greater than or equal to.

To check the GTE method, let's look inside the `modules/core/02-client/types/height.go` file.

```go:Go  modules/core/02-client/types/height.go {15-21,27}
// Compare implements a method to compare two heights. When comparing two heights a, b
// we can call a.Compare(b) which will return
// -1 if a < b
// 0  if a = b
// 1  if a > b
//
// It first compares based on revision numbers, whichever has the higher revision number is the higher height
// If revision number is the same, then the revision height is compared
func (h Height) Compare(other exported.Height) int64 {
	height, ok := other.(Height)
	if !ok {
		panic(fmt.Errorf("cannot compare against invalid height type: %T. expected height type: %T", other, h))
	}
	var a, b big.Int
	if h.RevisionNumber != height.RevisionNumber {
		a.SetUint64(h.RevisionNumber)
		b.SetUint64(height.RevisionNumber)
	} else {
		a.SetUint64(h.RevisionHeight)
		b.SetUint64(height.RevisionHeight)
	}
	return int64(a.Cmp(&b))
}

// GTE Helper comparison function returns true if h >= other
func (h Height) GTE(other exported.Height) bool {
	cmp := h.Compare(other)
	return cmp >= 0
}
```

`GTE()` compares each height and checks if it is greater than 0, so the actual comparison logic is in `Compare()`. The `Compare()` method first verifies the type, and if the Revision Numbers are the same, it compares the height, but if they are different, it only compares the Revision number.

Because of this logic, the abnormal transaction was previously able to pass the validation check.  
If the provided revision number is higher than the value stored in the light client, the `Compare()` function will always return 1, and `GTE()` will also return `true`. In the current logic, since the Revision number has a higher verification priority than the Block height, a timeout condition can be bypassed as long as the Revision number is high.

For example, assuming the current state of the counterparty chain is `1-10000`, if an attacker sends a packet with `9000-9100`, this packet will permanently never meet the timeout condition unless the counterparty chain's Revision number upgrades to 9999.

<Tip>
**Then can't we change the priority?**  
While looking at the code, I thought, "If comparing whether the Revision Numbers are not equal first is the problem, can't we just change the order of comparison?"

However, after thinking about it, the current logic is quite reasonable.  
Let's consider a situation where the counterparty chain's Revision Number increases due to an upgrade. Assume the current counterparty chain state stored in the Light Client is `1-900`, and the timeout height of the packet to be verified is `2-20`.

If the Revision Height were compared first, it would judge that the timeout has already passed since 900 > 20. However, in reality, since the counterparty chain just upgraded to Revision 2, `2-20` is a future point in time compared to `1-900`, but a normal packet would fail to pass.
![](/static/images/ibc-timeout-height-bug/revision-number-upgrade.png/)

When a chain goes through a hard fork or major upgrade, the Revision Number increases, but the Block Height can start again from a low value. In such a situation, we must check the Revision Number first to correctly determine which Revision number's block height it is.

Thus, I realized that prioritizing the Revision Number comparison is actually a reasonable design.

</Tip>

```go:Go  modules/apps/tranfer/client/cli/tx.go {65-70}
	cmd.Flags().Uint64(flagPacketTimeoutTimestamp, types.DefaultRelativePacketTimeoutTimestamp, "Packet timeout timestamp in nanoseconds from now. Default is 10 minutes. The timeout is disabled when set to 0.")
	cmd.Flags().Bool(flagAbsoluteTimeouts, false, "Timeout flags are used as absolute timeouts.")
	cmd.Flags().String(flagMemo, "", "Memo to be sent along with the packet.")
	flags.AddTxFlagsToCmd(cmd)

	return cmd
}

```

Looking inside the code, based on the `DefaultRelativePacketTimeoutHeight` value of `0-1000`, it adds the counterparty chain's revision number and height so that a timeout occurs 1000 blocks after the current one.

**In conclusion, starting from `v9.x.x` up to the latest version `v10.x.x`, `--packet-timeout-height` is blocked in the CLI, but up to `v8.x.x`, it was possible to set the flag.**

## 3-4 So is an attack also possible?

Having confirmed that it's theoretically possible, I tested whether an actual network attack was also possible.  
With the help of AI, I built a monitoring environment using Grafana and Prometheus in the PoC, and sent massive amounts of transactions multiple times via scripts, but contrary to my expectations, the network didn't flinch.

**From what I confirmed, although the transactions did not meet the timeout conditions, they were inherently normal transactions, so they were processed normally. To slow down the transaction processing speed, I tried putting massive amounts of strings in the `memo` field, but this also had no significant effect.**

> During this process, I realized that the IBC protocol is much more robust than I thought. Thanks to the structure where each chain verifies independently, I couldn't find an actual attack method at my knowledge level, and I thought, "There's a reason why it hasn't been hacked so far."

# 4. Contribution

Although I failed to implement an attack, I judged that the validity itself that could bypass the timeout was a problem.  
In particular, since many major chains use `v8.x.x` and below, an improvement seemed necessary.

To decide the direction of contribution, I looked at the issues and roadmap of ibc-go. During that process, I found that the team was preparing for the V2 version, and in V2, packet timeout height is not used at all, and timeouts are handled only with timestamp.  
Checking the [test code](https://github.com/cosmos/ibc-go/blob/4cb2ec7a329084f3664a3cbc6ecb3f4a04783498/modules/core/02-client/types/height_test.go#L16-L51), it seemed they were already aware of the problem that occurs when the Revision Number is high.

Upgrading to V2 fundamentally solves the problem, but since most chains in the production environment, including Osmosis, Celestia, and Milkyway, were still using versions `v8.x.x` or lower, and a V2 migration is not a simple library update, I thought it would take a considerable amount of time.

So I thought it was right to improve and raise this issue in the V1 version as well. Checking [CONTRIBUTING.md](https://github.com/cosmos/ibc-go/blob/4cb2ec7a329084f3664a3cbc6ecb3f4a04783498/CONTRIBUTING.md), it guided to register an Issue first before submitting a PR, so I summarized the findings and created an Issue.

> Security-related bugs are supposed to be reported via [HackerOne](https://hackerone.com/cosmos?type=team), but since configuring an actual attack scenario is difficult for this case and the CLI is blocked in the latest versions, the protocol's own verification mechanism still works. Therefore, I judged it to be closer to a simple bug rather than a security issue, and thought it was appropriate to proceed as an Issue.

In IBC, Light Client information is periodically renewed through Update client, and if it's not updated for a certain period, it enters an expired state and packets can no longer be sent.  
In other words, the Revision numbers of two chains communicating normally must always be the same.

Thinking that the verification logic could be improved based on this, I also attached improvement codes as an Issue comment.

The first is to pre-verify whether the Revision number of the Timeout height matches the current Revision number of the counterparty chain at the time of packet creation in SendPacket. The second is to immediately treat it as a timeout if the Revision numbers are different in the heightElapsed method.  
The first method is a proactive blocking method that prevents invalid packets from being created in the first place, and the second is a method to filter them out during the verification stage. I left the specific code implementation for each in the comments.
